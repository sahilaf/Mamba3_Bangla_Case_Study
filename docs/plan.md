# FINAL PLAN — State Tracking vs. Attention in Bangla: A Controlled Mamba-3 Case Study

**Status: feasibility verified 2026-07-16 (see [FEASIBILITY.md](FEASIBILITY.md)). Repo is ready to execute.**
**Target: MRL Workshop @ EMNLP 2026 — submission deadline Aug 10, 2026. Fallback: BLP Workshop (AACL).**

## 1. Research question (unchanged)

Mamba-3 (ICLR 2026) claims improved state-tracking via a more expressive recurrence,
complex/rotary state updates, and MIMO projections — validated on English-centric benchmarks.
Bangla verb morphology encodes person **and honorific register**, with long-distance
subject–verb dependencies and pro-drop discourse agreement.

> **Does Mamba-3's improved state tracking yield a measurable advantage over a
> parameter/token-matched Transformer on morphologically rich, low-resource Bangla — or does
> its fixed-size compressed state hurt under data scarcity?**

Either outcome is publishable; the question is untested for Bangla or any South Asian language.

## 3. Scope (final)

- **3 models**: Llama-style Transformer (RoPE + SwiGLU + RMSNorm, HF `LlamaForCausalLM`),
  **Mamba-3 SISO** (official block), and a **Hybrid** (Mamba-3 backbone with attention at
  2 of 15 layers, Samba/Jamba-style). The hybrid was added after seed-1 results showed a
  clean local-vs-distance trade-off; it tests whether a few attention layers recover local
  agreement while keeping the SSM's distance/discourse robustness. All matched on
  **non-embedding** parameters (~24.5M) and matched token budget.
- **1 seed** main runs; second seed only if units remain.
- **Token budget**: target 1.0B tokens (20 tok/param, Chinchilla-style), stretch 2.0B —
  finalized by the profiling script (`bangla_ssm/profile_throughput.py`) on the actual
  assigned GPU before launch.
- Colab Pro, CUDA only. Checkpoint/resume to Google Drive is built into the trainer from step 0.
- Framed as a pilot/short paper. MIMO variant + second seed are explicit stretch goals.

## 4. Evaluation suite (final)

1. **Held-out perplexity** on FineWeb-2 `ben_Beng` *test* split (never trained on).
2. **Custom Bangla minimal-pair probe suite** (the novel asset; generated by
   `bangla_ssm/probes/generate.py`, format-compatible with MultiBLiMP):
   - **SVA-distance**: person agreement (1/2-intimate/2-familiar/2-honorific/3-ordinary/
     3-honorific) with interveners binned by length (none/short/medium/long) — the
     state-tracking dial.
   - **Attraction**: interveners containing a competing person feature (e.g. 1st-person
     pronoun inside the intervener with a 3rd-person subject); the ungrammatical variant uses
     the lure's agreement — classic agreement-attraction, unstudied for SSMs in Bangla.
   - **Honorific register**: তিনি/উনি ↔ honorific verb form vs সে/ordinary NP ↔ plain form.
   - **Discourse honorific (pro-drop)**: register introduced in sentence 1, verb agreement
     tested across a sentence boundary with a dropped subject (and optional verbless filler
     sentence between) — a Bangla-specific long-range state probe with no English analogue.
   - Scoring: sum log-prob of full sentence, grammatical > ungrammatical ⇒ correct
     (BLiMP standard). Per-condition accuracy curves vs distance = the paper's headline figure.
3. **MultiBLiMP-ben (21 pairs)** as an external sanity check.
4. **Probe validation before the main runs**: run the suite on existing open Bangla-capable
   models of different strengths (e.g. a Bangla GPT-2, TituLM, an mGPT) — probes must separate
   strong from weak models and beat a unigram baseline, else revise.

## 5. Execution pipeline (matches the repo)

```
scripts/train_tokenizer.py   → 32k SentencePiece BPE (byte fallback) on a FineWeb-2 bn sample
scripts/pretokenize.py       → uint16 shards on Drive (train ~1–2.5B tok; test split → eval.bin)
scripts/count_params.py      → verify non-embedding param match; adjust n_layer if needed
bangla_ssm/probes/generate.py→ probe TSVs (CPU, local, free) + manual native-speaker pass
bangla_ssm/eval_minimal_pairs.py --hf_model ...  → probe validation on off-the-shelf models
bangla_ssm/profile_throughput.py → tokens/sec on assigned GPU → lock token budget
bangla_ssm/train.py          → toy run (both archs) → main runs, resumable across sessions
bangla_ssm/eval_ppl.py + eval_minimal_pairs.py → all numbers → paper/
```

## 6. Compute plan (Colab Pro)

- Profiling before committing: measure real tok/s; budget = tok/s × realistic session hours ×
  sessions, with ≥30% margin. Estimate: ~50M model @ 1B tokens ≈ **2–4 A100-hours per run**
  (T4 fallback ≈ 10× slower but still feasible across sessions).
- Trainer facts: deterministic resume (seeded window sampling keyed by step), atomic
  checkpoints every N steps to Drive (keep last 2), bf16 on A100/L4, fp16+GradScaler on T4.
- Budget rule: dev/toy on whatever GPU is assigned; main runs only on A100/L4. If units run
  low, finish the Transformer first (cheaper to re-run) and split runs across billing cycles.

## 8. Risks (updated)

- **Session disconnects** → resume is designed-in and tested in the toy phase. Loses minutes, not hours.
- **Slow GPU month** → profiling first; token budget set from measured throughput, not hope.
- **Probes non-discriminative** → validated on existing models *before* main runs (gate in §4.4).
- **Generated Bangla has errors** → conjugation tables are hand-written and unit-tested;
  still requires one native-speaker review pass of sampled items (you) — budgeted in days 3–7.
- **Single seed noise** → reported honestly as a pilot; probe suite has thousands of items so
  per-condition CIs are tight even with one seed; second seed is the first stretch goal.
- **`mamba-ssm` wheel mismatch on Colab** → install cell pins versions and falls back to
  source build; verified strategy in the driver notebook.
